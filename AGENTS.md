# Repository Guidelines

## Project Structure & Module Organization
- `src/`: TypeScript source for the MCP server.
- `src/auth/`: ONES login/session lifecycle.
- `src/discovery/`: endpoint/path discovery logic.
- Top-level modules in `src/` (for example `index.ts`, `ones-client.ts`, `config.ts`) contain server wiring and domain logic.
- `tests/`: Vitest suites, mostly mirroring `src/` paths (for example `tests/auth/session-manager.test.ts`) plus `tests/integration/` for flow-level checks.
- `scripts/`: smoke scripts for quick manual verification.
- `docs/plans/`: dated design/implementation notes (`YYYY-MM-DD-*.md`).
- `dist/`: build artifacts generated by TypeScript; do not edit manually.

## Build, Test, and Development Commands
- `npm install`: install dependencies.
- `npm run dev`: run the server with `tsx` against `src/index.ts`.
- `npm run build`: clean `dist/` and compile TypeScript.
- `npm run start`: run compiled output from `dist/src/index.js`.
- `npm run test`: run all tests once with Vitest.
- `node scripts/smoke.mjs "ONES"` or `node scripts/smoke.mjs "#12345"`: smoke-check search and doc retrieval paths.

## Coding Style & Naming Conventions
- Use TypeScript with `strict` mode and ESM modules.
- Follow existing formatting: 2-space indentation, double quotes, and semicolons.
- File names use `kebab-case` (for example `session-manager.ts`).
- Use `PascalCase` for exported types/classes and `camelCase` for functions/variables.
- In `src/`, keep relative runtime imports ESM-compatible (prefer `.js` extension in import paths).

## Testing Guidelines
- Framework: Vitest (`tests/**/*.test.ts`, Node environment).
- Name tests as `*.test.ts`; place tests under matching module paths when practical.
- Add or update tests for every behavior change, including integration coverage for ONES flow updates.
- No enforced coverage threshold is configured; reviewers expect meaningful regression protection.

## Commit & Pull Request Guidelines
- Follow Conventional Commit style used in history (`chore: ...`), and use `feat:`, `fix:`, `chore:` as appropriate.
- Keep commit subjects concise and imperative (for example `fix: retry login after 401`).
- PRs should include purpose/scope, linked issue or requirement, test evidence (`npm run test`), and docs/env updates when behavior or configuration changes.

## Security & Configuration Tips
- Never commit real ONES credentials; use local `.env` based on `.env.example`.
- Use least-privilege ONES accounts and avoid logging sensitive values.
